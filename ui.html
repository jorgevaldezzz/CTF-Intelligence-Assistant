QUERY
internal metadata service reachable from an unauthenticated /fetch?url= param, app fetches attacker-controlled URLs server-side
ANALYSIS

This is a textbook server-side request forgery setup 1. The unauthenticated url= parameter means the server itself is making the outbound request, so any URL you supply gets fetched with the server's network position and credentials — including the cloud metadata endpoint at 169.254.169.254 if this is hosted on AWS, GCP, or similar.

Start by probing whether the fetch follows redirects and whether it resolves internal hostnames. A documented pattern for this exact class is bypassing naive allowlists with alternate IP encodings (decimal, octal, or IPv6-mapped forms of the loopback/link-local range) 2. If the target validates the hostname but re-resolves it later (a TOCTOU gap), DNS rebinding is also worth trying.

A close real-world parallel: a CTF web challenge used almost this exact primitive to reach an internal admin API and pull a flag from a metadata-style endpoint 3. Worth reading end to end — the writeup's bypass technique for the URL parser is directly transferable here.
